# Plan: Add Kilo Agent + Fix Codex Backpressure Hang

## Problem Analysis

### 1. Kilo CLI Support

The project currently supports 4 agents (claude, codex, rovodev, opencode). Kilo Code CLI (`kilo`, installed via `npm install -g @kilocode/cli`) is a fork of OpenCode with the same underlying internals. It supports autonomous non-interactive mode via `kilo run --auto "message"`. Since Kilo is a fork of OpenCode, the agent implementation should follow the same HTTP server + SSE streaming pattern as the existing `OpenCodeAgent`.

### 2. Codex Backpressure Hang

The Codex agent implementation (`codex.ts`) spawns `codex exec --json ...` and reads stdout as JSONL. The bug: **stderr is collected by `setupChildProcessHandlers` but stdout is consumed by `parseJSONLStream` via Node's stream `.on('data')` which uses internal backpressure. When Codex produces large output, the internal buffer can fill up, causing the child process to block on write, resulting in a hang.**

The root cause is that `parseJSONLStream` attaches a `'data'` listener on stdout but does NOT properly handle backpressure. Node's Readable stream has an internal high-water mark (default 16KB). When the consumer processes data slower than the producer writes, the pipe backs up. Codex's Rust implementation uses bounded channels for stdout, and when the OS pipe buffer fills, the child process blocks.

Additionally, there's a **secondary issue**: the `--dangerously-bypass-approvals-and-sandbox` flag name may have changed in newer Codex versions (some reports reference `--dangerously-skip-permissions` style flags). This should be made configurable.

### Part 2: Fix Codex Backpressure Hang

#### 2.1 Root Cause

The `parseJSONLStream` function in `stream-utils.ts` attaches a `'data'` listener on the stdout stream. This uses Node's internal buffer management. When Codex writes large amounts of JSONL output faster than the callback can process it, the internal buffer fills. However, the real issue is more subtle:

**The actual hang is caused by stderr not being drained concurrently with stdout processing.** The `setupChildProcessHandlers` function attaches a stderr `'data'` listener, but the `'close'` handler only fires after both stdout and stderr have ended. If stdout processing blocks (due to the callback doing synchronous JSON.parse on large objects), or if the stream's internal buffer fills, the child process can deadlock.

**The fix**: Switch from event-based stream reading to a proper `for await...of` async iterator pattern that properly handles backpressure, OR ensure both stdout and stderr are consumed concurrently with proper flow control.

#### 2.2 `src/core/agents/stream-utils.ts`

- Modify `parseJSONLStream` to use async iteration with proper backpressure handling
- Add a new `parseJSONLStreamAsync` function that returns a Promise and properly awaits stream consumption
- The key change: use `stream.resume()` to put the stream into flowing mode (which handles backpressure correctly) OR switch to `for await (const chunk of stream)` pattern

**Actually, the simpler and more correct fix**: The current `parseJSONLStream` is synchronous and attaches a `'data'` handler. The issue is that when the Codex child process writes a lot of data to stdout, Node's internal buffer can fill. The `'data'` event approach should actually handle backpressure fine via `stream.pause()` / `stream.resume()`.

**Re-analyzing the actual Codex hang**: Based on the GitHub issues, the Codex hang is primarily caused by:

1. **MCP helper processes blocking startup** (issue #14470) â€” not fixable from gnhf side
2. **outputBytesCap backpressure in app-server** (issue #13821) â€” not directly applicable to `codex exec --json`
3. **The `--dangerously-bypass-approvals-and-sandbox` flag may not exist or may behave differently** in newer versions

**The actual fixable issue in gnhf's CodexAgent**: The current implementation does NOT consume stderr concurrently in a way that prevents pipe buffer overflow. While `setupChildProcessHandlers` does attach a stderr listener, the `'close'` event won't fire until the process exits â€” and if the process is blocked on writing to a full stdout pipe, it never exits.

**The real fix**: Switch the CodexAgent from the synchronous `parseJSONLStream` + `setupChildProcessHandlers` pattern to an async pattern that:

1. Consumes stdout via async iteration (proper backpressure)
2. Consumes stderr concurrently
3. Has a startup timeout to detect hangs early

- Use `for await...of` on stdout for proper flow control
- Drain stderr concurrently in a separate async task
- Make the `--dangerously-bypass-approvals-and-sandbox` flag conditional (check if codex version supports it, or make it configurable)
